# Workflow Feature

Профильный воркфлоу для задач с `[TASK_TYPE] = FEATURE`. Реализует последовательность стадий, результат каждой — артефакт-файл в папке задачи. Скилл получает уже резолвленный контракт от оркестратора и не пытается доразрешать параметры самостоятельно.

## 1. Контракт входа

Скилл вызывается `swift-toolkit:orchestrator`-ом через `Skill` tool с структурированными `args` в формате `key=value` (по одному на строку).

Структура полей описана в `swift-toolkit:orchestrator` (раздел **Outbound Contract**). Workflow-feature принимает все поля заполненными — invariant.

Если пришло пустое обязательное поле — workflow-feature не пытается восстановить, а возвращает `{status: error, reason: missing field <name>}` обратно в оркестратор.

Ключевые поля, которые непосредственно влияют на поведение этого воркфлоу:
- `start_stage`, `end_stage`, `stage_scope` — определяют, какие стадии исполнять.
- `start_phase` — точка входа внутри стадии (например, `Execute:phase=2.3`).
- `mode` — `manual` / `auto` (см. секции 3 и 4).
- `stack` — передаётся субагентам как контекст.
- `need_test`, `need_review` — управляют включением `swift-tester` и `swift-reviewer`.
- `archive_paths` — пути к уже сделанным бэкапам (бэкап делает оркестратор ДО вызова; workflow-feature их не создаёт).

## 2. Stages

- [ ] **Research** — консилиум: `swift-architect` + `swift-security` (через Task tool параллельно или последовательно по решению оркестратора). Артефакт: `Research.md` в папке задачи. Цель: исследовать предметную область, выявить риски, предложить варианты архитектуры.

- [ ] **Plan** — `swift-architect`. Артефакт: `Plan.md` с прогресс-таблицей фаз (см. State Detection в orchestrator: статусы ✅/🔄/⬜/⏸/🚫/⊘). План декомпозирует фичу на конкретные фазы и шаги.

- [ ] **Execute** — `swift-developer` + `swift-tester` (если `need_test=true` в args). Поэтапно реализует фазы из Plan.md, обновляя прогресс-таблицу после каждой фазы. Артефакты: исходный код в проекте + тесты.

- [ ] **Validation** — XcodeBuildMCP (`build_sim`, `test_sim`) + опционально mobile MCP (E2E сценарий ключевого пути). Проверка, что фича собирается и проходит тесты. Артефакт: `Validation.md` с логом и вердиктом.

- [ ] **Review** — `swift-reviewer` (если `need_review=true` в args). Артефакт: `Review.md`, **первая строка обязательно** `[REVIEW_STATUS] = APPROVED | CHANGES_REQUESTED | DISCUSSION` (это поле читается оркестратором/последующим воркфлоу для решения о auto-move).

- [ ] **Done** — финальный отчёт `Done.md`: что сделано, какие артефакты, валидация (статус сборки/тестов), возражения (если пользователь настоял на спорном решении).

## 3. Manual режим

После каждой завершённой стадии оркестратор задаёт пользователю `AskUserQuestion`: «<stage> готова. Перейти к следующей? [Yes / Edit / No]».

Workflow-feature **не делает `AskUserQuestion` сам** — он возвращает управление оркестратору после завершения стадии (см. секцию 5, Контракт выхода) с `next_recommended_action`. Решение о паузе, диалоге и обсуждениях, которые пишутся в `Questions.md`, — зона ответственности оркестратора.

Если хост-CLI не поддерживает `AskUserQuestion`, оркестратор использует текстовый fallback (нумерованные варианты + парсинг ответа). Это его ответственность, не workflow-feature.

## 4. Auto режим

Без пауз между стадиями. Workflow-feature последовательно проходит стадии в пределах `stage_scope` и возвращает финальный результат оркестратору одним выходом.

Единственное согласование, которое всегда требуется независимо от режима — финальный коммит (если в проекте принято коммитить). Это тоже ответственность оркестратора, не workflow-feature.

## 5. Контракт выхода

После каждой стадии (в `manual` режиме) или после полного прохода (в `auto` режиме) workflow-feature возвращает в оркестратор JSON-подобную структуру:

```
{
  status: ok | error | interrupted,
  last_completed_stage: Research | Plan | Execute | Validation | Review | Done,
  artifact_path: <путь к финальному артефакту, например Tasks/ACTIVE/001-feature/Done.md>,
  next_recommended_action: continue | stop | ask_user,
  notes: <свободный текст, опциональный>
}
```

Семантика полей:
- `status=ok` — стадия завершена корректно; `status=error` — произошла ошибка (включая «empty required field»); `status=interrupted` — выполнение было прервано (например, пользователь сказал «No» в AUQ оркестратора, и тот передал прерывание дальше).
- `last_completed_stage` — последняя реально завершённая стадия (не та, на которой остановились с ошибкой).
- `artifact_path` — путь к ключевому артефакту последней стадии (`Research.md`, `Plan.md`, `Validation.md`, `Review.md`, `Done.md`).
- `next_recommended_action=continue` — можно сразу запускать следующую стадию; `stop` — естественный финиш (Done) или фатальная ошибка; `ask_user` — нужно подтверждение пользователя перед продолжением (например, после Review с `CHANGES_REQUESTED`).
- `notes` — короткое свободное описание (например, «build_sim failed: SwiftCompile error in FooView.swift:42»).

На основании этого оркестратор решает: продолжить, прервать, спросить пользователя.

## 6. Что workflow-feature НЕ делает

- НЕ маршрутизирует — выбор профиля сделан в оркестраторе до вызова.
- НЕ читает `Task.md` для определения стека/режима — всё пришло в `args`.
- НЕ дёргает `task-new` или `task-move` — это не его зона.
- НЕ принимает решения о пропуске стадий — оркестратор уже передал `start_stage`, `end_stage`, `stage_scope`.
- НЕ создаёт бэкапы в `_archive/` — это сделал оркестратор до передачи управления; пути уже в `archive_paths`.
- НЕ задаёт `AskUserQuestion` — это делает оркестратор между стадиями в `manual` режиме.
- НЕ согласует коммит с пользователем — этим занимается оркестратор после возврата `next_recommended_action`.
